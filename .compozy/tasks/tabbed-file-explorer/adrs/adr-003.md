# ADR-003: Extend the shared session tab model to support file tabs and terminal tabs

## Status

Accepted

## Date

2026-05-29

## Context

Another ADE already uses a single session-scoped tab model for terminal work. `WorkspaceStore`, `WorkspaceTab`, `DefaultWorkspaceCommandService`, `RestoreSnapshot`, and the SQLite `tabs` table all assume one ordered tab namespace per session. The user selected a shared session tab strip for MVP instead of a separate file-tab strip or a single-file editor surface.

The technical design must add file tabs without creating a second open-item system that duplicates selection, ordering, close behavior, and restore logic. The current architecture is terminal-specific, so the main question is whether to generalize the existing tab model or create a parallel file-tab model.

## Decision

The implementation will extend the existing shared session tab model so terminal tabs and file tabs coexist in the same ordered tab strip.

This means:

- `WorkspaceTab` will be generalized to represent multiple tab kinds.
- File-tab metadata will be stored in the same persistence and restore pipeline as terminal tabs.
- Selection, ordering, activation timestamps, close behavior, and restore ordering will continue to use the existing tab ID namespace.
- The UI tab chrome will render different labels, icons, and close rules based on tab kind.

## Alternatives Considered

### Alternative 1: Separate file tab strip inside the editor area

- **Description**: Keep the current terminal tab model unchanged and add a second file-tab model inside a new editor panel.
- **Pros**:
  - Minimizes change to the current terminal-tab data model.
  - Keeps terminal workflows visually distinct.
  - Makes it easier to prototype the editor panel in isolation.
- **Cons**:
  - Creates two competing tab systems in one workspace.
  - Duplicates selection, ordering, close, and restore mechanics.
  - Increases user confusion about which tab strip owns the active context.
- **Why rejected**: It adds needless structural complexity and weakens the shared session mental model.

### Alternative 2: Single-file editor surface with no file tabs

- **Description**: Open one file at a time in the editor area and avoid file tabs in MVP.
- **Pros**:
  - Smallest data-model change.
  - Simplifies restore and close logic.
  - Avoids mixed terminal/file tab ordering.
- **Cons**:
  - Conflicts with the approved PRD experience for tabbed files.
  - Makes file comparison and fast back-and-forth editing weaker.
  - Reduces reuse of the existing tab investment.
- **Why rejected**: It under-delivers on the approved product flow and gives up useful reuse of existing tab mechanics.

## Consequences

### Positive

- Reuses the existing selection, ordering, close, and restore pipeline.
- Keeps session context coherent across terminal and file work.
- Avoids a second open-item model and a second restore system.
- Makes future mixed workflows easier because all session tabs share the same identity and ordering model.

### Negative

- Generalizing `WorkspaceTab` requires schema, UI, and command-service changes across multiple layers.
- Existing terminal-specific labels and assumptions must be revised carefully.
- Close behavior becomes more nuanced because file tabs and terminal tabs do not share the same runtime semantics.

### Risks

- **Over-generalization**: the team may introduce a wider abstraction than MVP needs.  
  Mitigation: add only the minimum fields and enum cases required for terminal and file tabs.
- **Terminal regressions**: terminal workflows could break while generalizing the shared model.  
  Mitigation: preserve current terminal behavior as the default path and extend tests around selection, order, and close flow.
- **Restore confusion**: mixed tab kinds may restore in surprising ways if ordering is not preserved exactly.  
  Mitigation: continue using a single tab-order snapshot and cover mixed-tab restore cases with integration tests.

## Implementation Notes

- Add a tab-kind discriminator and file-specific metadata to the existing tab model instead of creating new top-level tab entities.
- Keep terminal runtime state outside `WorkspaceStore`, and apply the same split for live editor buffer state.
- Update the tab chrome to display per-kind titles and icons while keeping a single selection model.
