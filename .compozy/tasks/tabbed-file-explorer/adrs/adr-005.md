# ADR-005: Persist file-tab metadata only, with explicit save and no unsaved-buffer restore

## Status

Accepted

## Date

2026-05-29

## Context

Another ADE already treats restore as metadata continuity rather than deep runtime recovery. The current terminal restore flow reopens tabs and recreates fresh terminal surfaces, not live process state. For file editing, the user selected explicit save with dirty state and chose an MVP restore model that reopens file tabs after relaunch but discards unsaved buffers.

The technical design must decide whether to persist unsaved editor buffers, autosave changes, or keep persistence limited to saved file metadata and reload from disk when needed.

## Decision

The MVP will persist file-tab metadata only and will restore reopened file tabs from disk. Unsaved editor buffers will not be persisted across relaunch, and saving will be explicit with visible dirty-state tracking.

This means:

- Persist file path, tab ordering, and selected-tab context in the shared workspace metadata store.
- Keep live editor buffer contents and dirty state in runtime-only memory.
- On restore, recreate file tabs and reload their saved contents from disk.
- If the app closes before a save, unsaved edits are discarded.

## Alternatives Considered

### Alternative 1: Restore full editor state including unsaved buffers

- **Description**: Persist open file text, cursor position, and unsaved changes as part of restore.
- **Pros**:
  - Feels more like a full editor.
  - Minimizes data loss across interruptions.
  - Gives the most continuity for editing sessions.
- **Cons**:
  - Expands the persistence model substantially.
  - Introduces more privacy, storage, and conflict-handling complexity.
  - Breaks the current product pattern of metadata-first restore.
- **Why rejected**: Too large and too stateful for the MVP continuity promise.

### Alternative 2: Autosave changes on focus change or continuously

- **Description**: Persist edits to disk automatically during or after editing rather than requiring explicit save.
- **Pros**:
  - Reduces the chance of lost edits.
  - Simplifies relaunch behavior because disk already reflects current text.
  - Aligns with some modern editor defaults.
- **Cons**:
  - Raises the risk of unintended writes in a quick-fix editor.
  - Makes terminal-adjacent editing feel more opaque.
  - Changes the trust model before users learn the file surface.
- **Why rejected**: Explicit save is safer and more predictable for the first release.

## Consequences

### Positive

- Aligns file restore behavior with the app’s existing metadata-first continuity model.
- Limits persistence complexity and avoids storing unsaved source contents.
- Makes file writes deliberate, which matches the trust goals of the product.
- Keeps runtime editor state separate from durable workspace metadata.

### Negative

- Users can lose unsaved quick edits if they close or relaunch before saving.
- Dirty-state UX must be clear and reliable.
- Restore behavior is less convenient than a full editor would provide.

### Risks

- **User surprise about unsaved changes**: users may expect stronger buffer continuity once editing exists.  
  Mitigation: show clear dirty indicators, close confirmation for unsaved file tabs, and explicit save actions.
- **Disk divergence**: external edits may change files between open and restore.  
  Mitigation: reload from disk on restore and provide simple revert/reload behavior in the editor flow.
- **Mixed close semantics**: terminal close checks and file dirty-state checks differ.  
  Mitigation: route close decisions through the shared command service with per-tab-kind rules.

## Implementation Notes

- Extend close flows so file tabs prompt on unsaved changes while terminal tabs keep process-close protection.
- Persist only stable file metadata needed to reconstruct tabs after restore.
- Reuse existing restore diagnostics patterns for missing or inaccessible files.
