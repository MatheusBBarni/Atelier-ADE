# ADR-004: Use CodeEditorView for MVP syntax-aware editing with highlighting and editing only

## Status

Accepted

## Date

2026-05-29

## Context

The PRD requires syntax-aware file tabs and quick-fix editing inside Another ADE. The user explicitly requested `CodeEditorView` for the editor surface. Library research showed that `CodeEditorView` is a SwiftUI TextKit 2-based editor with syntax highlighting, editing state bindings, and optional diagnostics/completion hooks, but it is also described as pre-release quality.

The technical question is not whether to add a full language-service stack in MVP. It is how much of the editor library to wire up without turning the first release into a broad editor platform.

## Decision

The MVP will use `CodeEditorView` as the embedded editor surface, limited to syntax highlighting, selection, scrolling, and direct text editing.

This means:

- Add `CodeEditorView` and `LanguageSupport` as Swift package dependencies.
- Configure language-aware highlighting for supported source-file types.
- Defer diagnostics, completion, hover, and external language-service integration.
- Treat the editor as a narrow quick-fix surface, not an editor platform.

## Alternatives Considered

### Alternative 1: Native SwiftUI/TextKit editor assembled in-house

- **Description**: Build the editor surface directly from built-in Apple editor primitives.
- **Pros**:
  - No third-party dependency risk.
  - Full control over behavior and future evolution.
  - Easier to keep the codebase dependency graph minimal.
- **Cons**:
  - Slower path to syntax-aware editing.
  - Higher implementation and polish cost for MVP.
  - More custom editor behavior to test and maintain.
- **Why rejected**: It spends too much effort recreating capabilities the selected library already provides for MVP.

### Alternative 2: CodeEditorView with diagnostics and language-service hooks from day one

- **Description**: Use the same library but wire messages, diagnostics, and future language-service surfaces immediately.
- **Pros**:
  - Creates a richer editor foundation sooner.
  - Leaves room for future authoring features.
  - Makes the editor feel more capable early.
- **Cons**:
  - Pulls MVP toward richer IDE behavior.
  - Adds integration and runtime complexity that the PRD does not require.
  - Introduces more moving parts around performance and testing.
- **Why rejected**: MVP only needs syntax-aware quick-fix editing, not a broader language-service surface.

## Consequences

### Positive

- Speeds delivery of a syntax-aware editor that matches the approved PRD.
- Keeps the implementation narrowly aligned to the MVP editing promise.
- Avoids premature investment in completion, diagnostics, and hover systems.
- Leverages a purpose-built SwiftUI editor rather than building one from scratch.

### Negative

- Introduces a third-party dependency that the app does not currently use.
- Carries pre-release library risk and possible API churn.
- Limits the MVP to token-based highlighting rather than richer semantic behavior.

### Risks

- **Dependency instability**: upstream changes or edge cases could affect editor behavior.  
  Mitigation: isolate integration behind a narrow wrapper view/controller and keep MVP features limited.
- **Feature pressure**: availability of diagnostics and completion hooks may encourage scope creep.  
  Mitigation: do not expose those capabilities in the MVP command surface or UI.
- **Performance surprises**: narrow-right-pane editing may need custom layout tuning.  
  Mitigation: keep minimap and non-essential editor features disabled in MVP and optimize for common source files.

## Implementation Notes

- Use the library as a view-layer dependency only; do not let editor-specific types spread across core workspace models.
- Map supported file types to the minimum language configurations needed for syntax coloring.
- Keep library configuration centralized so later language-service expansion remains optional.
