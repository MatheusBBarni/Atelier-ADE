# ADR-001: Scope tabbed-file-explorer as a session-aware working-set navigator

## Status

Accepted

## Date

2026-05-29

## Context

Another ADE is currently a native macOS, local-first, terminal-first workspace built around projects, sessions, and terminal tabs. The current PRD at `.compozy/tasks/native-mac-ade/_prd.md` explicitly lists embedded editor or full IDE behavior as out of scope for V1.

The new idea asks for a right sidebar that shows workspace folders and files, and opens clicked files in new tabs with file content in an editor. User clarification narrowed the job to faster navigation for active coders during active coding, with success defined as fewer context switches.

Research showed that file explorers and tabbed file access are table stakes in modern coding tools, but they are not strong differentiation on their own. The codebase also shows that Another ADE has no existing file tree, file editor, or generic document-tab model today. Its current architecture is terminal-tab oriented, and restore/persistence flows assume project -> session -> terminal tab. A broad “file explorer + editable editor tabs” move would therefore introduce new load-bearing concepts including file access, open-item identity, restore semantics, preview/edit boundaries, and file-state expectations.

The council debate converged on a narrower framing: the highest-value V1 is not a mini-IDE, but a session-aware working-set navigation layer that reduces context switching while preserving the terminal-first wedge.

## Decision

For V1, `tabbed-file-explorer` will be scoped as a **session-aware working-set navigator**, not as a broad embedded editor feature.

This means:

- The right sidebar should primarily optimize for fast access to relevant files during an active session.
- The primary mental model should be a **working set**: recent, pinned, or context-relevant files, with a recognizable file tree available as a secondary or fallback affordance.
- If files open inside Another ADE in V1, they should open as **lightweight file-context or read-only preview tabs**, not full editable editor tabs.
- External editor handoff should remain available as an explicit escape hatch.
- Full editable document behavior, IDE-style file ownership, and broad editor expectations remain out of scope for this phase.

## Alternatives Considered

### Alternative 1: Full right-sidebar file explorer with editable editor tabs

- **Description**: Add a conventional IDE-style tree plus full file editor tabs inside Another ADE.
- **Pros**:
  - Matches mainstream user expectations immediately.
  - Removes more context switches in the ideal case.
  - Could make Another ADE feel more complete as a standalone coding environment.
- **Cons**:
  - Directly conflicts with the current V1 wedge and PRD non-goals.
  - Requires new editor, file, restore, and tab architecture that does not exist today.
  - Creates broad expectations around editing, saving, dirty state, search, and file watching.
- **Why rejected**: It expands the product into mini-IDE territory before the terminal-first wedge is proven and before the architecture is ready for that contract.

### Alternative 2: External-editor handoff only

- **Description**: Add sidebar navigation, but always open files in an external editor instead of inside Another ADE.
- **Pros**:
  - Lowest architectural risk.
  - Preserves a strict terminal-first product boundary.
  - Avoids accidental editor-contract creep.
- **Cons**:
  - Under-delivers on the “fewer context switches” promise.
  - Weakens the value of adding an in-app sidebar for active coding.
  - Risks validating only file discovery, not in-session file context flow.
- **Why rejected**: It is a useful fallback or escape hatch, but too weak as the primary product move for this idea.

### Alternative 3: Conventional file tree as the primary V1 surface with preview tabs

- **Description**: Ship a recognizable file tree as the main sidebar model, with lightweight preview tabs for opened files.
- **Pros**:
  - Easier for users to understand immediately.
  - Closer to mainstream editor ergonomics.
  - Supports navigation without full editing.
- **Cons**:
  - Keeps the product mentally anchored on filesystem hierarchy rather than session relevance.
  - Risks recreating generic IDE behavior with limited differentiation.
  - Makes it easier for the tree to become the product center of gravity.
- **Why rejected**: The council preferred a stronger differentiated framing where the working set is primary and the tree is secondary.

## Consequences

### Positive

- Preserves the terminal-first wedge while still addressing a real, high-frequency navigation problem.
- Creates a path to validate reduced context switching without committing to full embedded editing.
- Keeps the first version focused on the differentiated job: session-aware navigation during active coding.
- Allows a recognizable file tree for trust and orientation without making it the governing product model.

### Negative

- Some users will still expect editable tabs and may perceive the first release as incomplete.
- The product must communicate the preview/context boundary clearly to avoid confusion.
- This scope still introduces new file-access and open-item concepts that the current codebase does not yet model.

### Risks

- **Expectation creep**: preview tabs may trigger requests for full editing quickly.  
  Mitigation: label and constrain preview behavior clearly; keep external-editor handoff prominent.
- **Architectural coupling**: a fast UI implementation could accidentally entangle file previews with terminal-tab models.  
  Mitigation: define explicit file/open-item boundaries before implementation.
- **Wedge dilution**: the sidebar could drift toward generic IDE behavior.  
  Mitigation: prioritize session relevance, working-set views, and context-based navigation over generic tree completeness.
- **File access and restore complexity**: project-path and bookmark handling may become fragile if expanded casually.  
  Mitigation: treat file access, bookmark handling, and restore semantics as explicit scope constraints for V1.

## Implementation Notes

This ADR is a product-scope decision, not a full technical design. The later PRD and TechSpec should preserve these boundaries:

- working set first, tree second
- preview/read-only file context before editable tabs
- external editor handoff retained
- no broad embedded editor or full IDE commitment in this phase
