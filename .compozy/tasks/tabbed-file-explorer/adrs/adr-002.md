# ADR-002: Adopt a working-set-first quick-fix editor approach for the PRD

## Status

Accepted

## Date

2026-05-29

## Context

The idea phase established `tabbed-file-explorer` as a session-aware working-set navigator with a secondary file tree. During PRD brainstorming, the user clarified that the feature should go beyond inspection alone: files should open in-app, display syntax-aware coloring, and support quick edits without forcing the user into an external editor for every small change.

This creates a product tension. Another ADE is still positioned as a terminal-first, session-first macOS workspace, and the current product language explicitly avoids becoming a full IDE too early. Market research shows that users expect file trees, tabs, preview behavior, and strong navigation ergonomics, but they do not necessarily require broad editor parity in the first version. The PRD therefore needs an approach that improves daily usefulness while protecting the current wedge.

## Decision

The PRD will use a **working-set-first quick-fix editor** approach for MVP.

This means:

- The right sidebar remains a session-aware working set first and a file tree second.
- Files open inside Another ADE in syntax-aware tabs.
- The first release supports quick fixes and light code editing in context.
- The first release does not promise broad editor-first behavior or parity with general-purpose IDEs.
- The product continues to frame the feature as a faster session workflow, not a full coding suite.

## Alternatives Considered

### Alternative 1: Navigation-first inspector

- **Description**: Add the sidebar and in-app inspection flow first, while keeping editing minimal or deferred.
- **Pros**:
  - Lowest scope and lowest adoption risk.
  - Keeps the terminal-first wedge very clear.
  - Avoids broad expectations around editing behavior.
- **Cons**:
  - Under-serves the user request for in-app editing.
  - Leaves many small code changes still dependent on an external editor.
  - May feel incomplete relative to user expectations once files open in tabs.
- **Why rejected**: Too conservative for the validated user need, which now includes small in-context edits.

### Alternative 2: Editor-forward sidebar workspace

- **Description**: Make the sidebar, file tabs, and stronger multi-file editing the center of the experience.
- **Pros**:
  - Highest immediate in-app utility for coding work.
  - Closer to mainstream IDE expectations.
  - Could reduce external-tool switching more dramatically in the best case.
- **Cons**:
  - Pulls the product toward a mini-IDE too early.
  - Raises user expectations around broad editing capabilities, not just quick fixes.
  - Increases risk of diluting the terminal-first product story.
- **Why rejected**: Too broad for the MVP and too likely to overwhelm the current wedge before it is proven.

## Consequences

### Positive

- Better matches the user’s explicit request for in-app edits and syntax-aware file tabs.
- Keeps the MVP valuable for real coding sessions, not only inspection.
- Preserves a clear product narrative around quick, in-context work rather than full editor replacement.
- Creates a stronger daily-use habit opportunity than inspection-only scope.

### Negative

- Some users may expect more editor depth than the MVP intends to provide.
- Product messaging must clearly distinguish quick-fix editing from a broad editor promise.
- The PRD will need sharper non-goals to prevent scope creep.

### Risks

- **Expectation creep**: users may assume full IDE behavior once editing is present.  
  Mitigation: define and communicate a narrow editing promise in the PRD and rollout plan.
- **Wedge dilution**: the feature could start to dominate the product identity.  
  Mitigation: keep the working set and session flow as the primary framing.
- **Scope expansion pressure**: richer editing features may look like natural follow-ups before MVP proof exists.  
  Mitigation: defer editor-heavy expansions to later phases with explicit success gates.

## Implementation Notes

This ADR defines product scope only. The follow-on TechSpec may choose the technical solution, but the PRD should preserve these product boundaries:

- working set first
- recognizable tree second
- syntax-aware file tabs
- quick-fix editing, not editor parity
- external editor still available when users need more power
