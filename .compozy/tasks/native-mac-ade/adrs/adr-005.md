# ADR-005: Persist Metadata-Only State and Restore Fresh Shells

## Status

Accepted

## Date

2026-05-28

## Context

The PRD defines V1 as workflow-speed-first and explicitly limits continuity to basic restore rather than deep checkpoints or full historical replay. The user chose a relaunch model that restores metadata and reopens fresh shells, and a persistence scope limited to projects, sessions, tabs, and restore snapshots.

The implementation must therefore decide how much terminal state to persist, whether live shells should survive app relaunch, and how SQLite should be used without drifting into a deeper continuity platform.

## Decision

V1 will persist metadata-only state in SQLite and will restore fresh shell processes on relaunch rather than preserving live shells.

Persisted records will cover:

- projects and their display metadata
- sessions and their names, ordering, and project ownership
- tabs and their session ownership, working-directory intent, and launch profile
- restore snapshots capturing selected project, selected session, and open tab layout
- lightweight agent shortcut definitions and launch arguments when needed

On relaunch, the app will rebuild the visible workspace from that metadata and create fresh terminal surfaces and fresh terminal processes for restored tabs.

## Alternatives Considered

### Alternative 1: Preserve Live Shells Across Relaunch

- **Description**: Keep terminal processes alive independently and reconnect the app to them after restart.
- **Pros**: Stronger continuity; closer to true session persistence; fewer surprises for long-running shells.
- **Cons**: Adds a heavier runtime model, more lifecycle complexity, and more failure cases than V1 needs.
- **Why rejected**: The product direction explicitly chose basic restore over deep continuity for the first release.

### Alternative 2: Add Recent Activity or Checkpoint Records in V1

- **Description**: Persist recent activity summaries or checkpoint/milestone records alongside restore state.
- **Pros**: Better orientation after returning; stronger trust story.
- **Cons**: Expands the data model, UI, and restore semantics before the basic workflow loop is proven.
- **Why rejected**: V1 should only store the minimum data needed for navigation and restore.

## Consequences

### Positive

- Keeps persistence simple and closely aligned with the PRD.
- Avoids background session management or hidden process lifetimes.
- Makes restore behavior predictable and easier to test.
- Limits stored data to what is needed for V1 navigation and relaunch.

### Negative

- Users will not recover live terminal processes after app restart.
- Scrollback and transient shell state will be lost unless explicitly re-created by the user.
- The app must communicate restore semantics clearly to avoid false expectations.

### Risks

- **Users expect stronger continuity than provided** — Mitigate with explicit product language and UI cues that restoration reopens context, not live shells.
- **Metadata model grows opportunistically** — Mitigate by keeping SQLite tables tightly scoped to project/session/tab/restore concepts.
- **Relaunch feels incomplete for some workflows** — Mitigate by designing the restore flow to reopen the right project, session, and launch profile with minimal friction.

## Implementation Notes

Secrets remain outside SQLite. If session shortcuts require sensitive values later, they should be stored in Keychain and referenced indirectly from metadata.
