# ADR-003: Use a SwiftUI App Shell with AppKit Terminal Hosting

## Status

Accepted

## Date

2026-05-28

## Context

The repository is greenfield. There is no existing macOS app scaffold, module structure, or terminal integration code. Product ADRs already fix the V1 shape around a local-first project -> session -> tab model, and the implementation target is a native macOS app built with Swift and SwiftUI while embedding Ghostty terminal surfaces.

Ghostty's official macOS integration path is AppKit `NSView` hosting with a SwiftUI-based application shell. The host application is responsible for tabs, window restoration, close behavior, and terminal surface lifecycle. The technical question is how much architecture to introduce in V1 while preserving native fit and keeping the session-first model coherent.

## Decision

V1 will use a SwiftUI app shell for scenes, sidebar navigation, and high-level state presentation, paired with AppKit-backed terminal host controllers and views for embedded Ghostty surfaces.

The application boundary will be:

- SwiftUI scenes and feature views for project list, session list, tab chrome, and app-level navigation
- AppKit terminal host components for `NSView`/controller lifecycle, focus, sizing, and surface attachment
- local command services for core actions such as open project, create session, create tab, restore state, and run session shortcuts
- lightweight feature stores/view models that coordinate SwiftUI state with command services and persistence

This keeps the product aligned with upstream Ghostty patterns while avoiding an over-designed multi-package architecture before the first app scaffold exists.

## Alternatives Considered

### Alternative 1: Mostly AppKit

- **Description**: Build the product primarily with AppKit controllers and views, using SwiftUI only for lightweight wrappers.
- **Pros**: Natural fit for `NSView`-based terminal hosting; fewer UI boundary crossings.
- **Cons**: More boilerplate for common app UI; weaker fit with the intended SwiftUI-first product shell; slower for routine product iteration.
- **Why rejected**: The app needs rich sidebar and state-driven desktop UI, and SwiftUI is the better default shell for that work.

### Alternative 2: Heavier Modular Split From Day One

- **Description**: Start with a more layered or package-oriented architecture before implementation begins.
- **Pros**: Stronger future separation; easier large-scale refactors later.
- **Cons**: Higher setup cost; more abstractions than V1 needs; greater risk of architecture-first churn in a greenfield repo.
- **Why rejected**: V1 needs a clear, minimal architecture that matches the product shape without inventing premature structure.

## Consequences

### Positive

- Aligns the app shell with the intended SwiftUI product experience.
- Uses the integration model Ghostty itself follows on macOS.
- Keeps the project -> session -> tab boundary explicit in UI and state ownership.
- Preserves YAGNI by avoiding unnecessary module or package expansion.

### Negative

- Introduces a mixed SwiftUI/AppKit stack that requires careful boundary management.
- Terminal-focused concerns such as focus, sizing, and teardown live outside the pure SwiftUI world.
- Some UI state will need translation between feature stores and AppKit terminal controllers.

### Risks

- **Boundary leakage between SwiftUI and AppKit** — Mitigate by isolating Ghostty hosting inside dedicated terminal host components.
- **Main-thread lifecycle errors** — Mitigate by enforcing main-actor ownership for terminal surface creation, focus, and teardown.
- **Premature service sprawl** — Mitigate by keeping command services narrow and tied to concrete V1 workflows only.

## Implementation Notes

The initial scaffold should define a small number of app-local layers: app shell, feature stores/views, terminal host, persistence, and command services. No extra packages are required for V1.
