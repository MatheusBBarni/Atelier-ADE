# ADR-001: Session-First Native macOS ADE Scope for V1

## Status

Accepted

## Date

2026-05-28

## Context

The product idea is a native macOS agentic development environment built with Swift/SwiftUI and Ghostty for terminal rendering. The target user is a solo, AI-heavy developer who frequently switches across repositories and agent sessions on one Mac. The current workaround is an IDE plus terminal tabs, which creates context-switching overhead and weak session continuity.

Market research showed strong AI-tool adoption, high tool-switching pain, and growing interest in terminal-first agent workflows. Orca is the closest competitor, but it competes on breadth: worktrees, multi-agent orchestration, review loops, SSH, browser tooling, and wider operational surfaces. The greenfield repo has no existing architecture constraints.

The core decision is whether V1 should compete on breadth, stay narrowly session-first, or add a thin agent-aware layer while remaining local-first and native.

## Decision

V1 will be positioned as a native macOS control center for terminal agents, not as a broad AI IDE or Orca-style orchestration platform.

The recommended V1 scope is a complete, local-first session spine:

- persistent opened projects in a sidebar
- per-project sessions with default timestamp naming and rename support
- terminal tabs that open in the selected project's folder by default
- durable restore of projects, sessions, and tabs after relaunch
- clear active project/session state and lightweight trust cues such as visible session state, last activity, and simple resume/recovery
- local persistence in SQLite for metadata only
- secrets stored outside SQLite, using Keychain when needed
- explicit user initiation for project-opening and any cross-project action
- no hidden background execution model in V1

If agent-specific affordances are included in V1, they must be expressed inside the session model as lightweight templates or commands, not as a new top-level orchestration abstraction.

## Alternatives Considered

### Alternative 1: Broader Orca-style ADE V1

- **Description**: Include worktrees, multi-agent orchestration, richer automation, remote flows, and broader IDE-like surfaces in the first release.
- **Pros**: More directly comparable to Orca; potentially higher ceiling; may satisfy advanced operator needs earlier.
- **Cons**: Much larger complexity budget; more state to manage; greater support burden; weaker focus; slower validation cycle.
- **Why rejected**: It sacrifices the main wedge—native simplicity, trust, and calm session management—before the core workflow replacement hypothesis is proven.

### Alternative 2: Minimal Project-and-Tab Wrapper

- **Description**: Ship only project sidebar, basic sessions, and tabs, with little state visibility, recovery, or trust-oriented supervision.
- **Pros**: Fastest path to shipping; smallest implementation surface.
- **Cons**: Risks feeling like a polished launcher; may not earn trust for serious agent work; weak differentiation from existing terminals and IDE tabs.
- **Why rejected**: The council judged supervision, recovery, and state clarity to be part of the core product value, not optional polish.

### Alternative 3: Agent-Aware Top-Level Abstractions in V1

- **Description**: Add first-class agent entities, agent-specific navigation, or orchestration surfaces alongside projects and sessions.
- **Pros**: Stronger agent identity; clearer purpose-built positioning.
- **Cons**: Creates hidden coupling; expands persistence and lifecycle complexity; pressures the app toward orchestration scope too early.
- **Why rejected**: Agent support should live within sessions in V1 so the project -> session -> tab model remains the load-bearing boundary.

## Consequences

### Positive

- Sharpens product positioning against Orca through native feel, lower complexity, and session clarity.
- Maximizes chances of proving the workflow replacement KPI quickly.
- Keeps the architecture coherent around project -> session -> tab.
- Builds trust through restore, visible state, explicit boundaries, and proportionate security controls.

### Negative

- Advanced users may find V1 underpowered compared with Orca or broader ADEs.
- Some desirable agent workflows will remain manual until later versions.
- The product may still be perceived as "just a terminal wrapper" if session trust cues are too weak in execution.

### Risks

- **Underbuilt operator control** — Mitigate with visible session state, lightweight recovery, resume, and provenance cues in V1.
- **Scope creep toward orchestration** — Mitigate by rejecting any feature that introduces a new top-level abstraction outside project, session, or tab.
- **Security trust gap** — Mitigate with metadata-only SQLite, Keychain-backed secrets, explicit user initiation, and no hidden background execution.
- **Weak differentiation** — Mitigate by emphasizing native Mac quality, correct cwd behavior, restore reliability, and all-day usability.

## Implementation Notes

This ADR sets product and scope boundaries for the idea phase. It does not define implementation architecture beyond enforcing the session-first model and local-first trust constraints.
