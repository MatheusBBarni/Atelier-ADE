# ADR-002: Adopt a Workflow-Speed-First PRD for V1

## Status

Accepted

## Date

2026-05-28

## Context

The product concept is a native macOS agentic development environment for solo, terminal-heavy developers who switch between a few important repositories many times per day. The existing idea and ADR-001 establish the main wedge: a local-first, session-first product that is narrower and calmer than Orca-style orchestration.

PRD clarification confirmed that V1 should optimize for all three core user outcomes—resume work instantly, switch projects confidently, and supervise agent activity—but the non-negotiable differentiator is faster project-based workflow. The user also chose a narrower continuity promise for V1: basic restore rather than deeper checkpoint-oriented memory. V1 should best serve a workflow with a small set of active projects and frequent returns, and agent-specific behavior should remain lightweight.

The remaining decision is how the PRD should frame V1: lead with faster workflow, lead with deeper continuity, or lead with stronger supervision.

## Decision

The PRD will adopt a workflow-speed-first approach for V1.

This means the first release will focus on the fastest credible path to replacing an IDE-plus-terminal-tabs workflow for solo Mac developers:

- persistent project sidebar as the main navigation surface
- project-scoped sessions that are easy to start, find, rename, and revisit
- new tabs opening in the selected project's context by default
- basic restore of projects, sessions, and tabs after relaunch
- lightweight agent shortcuts that help users start common session types without changing the core product model

Session supervision and continuity remain important, but in V1 they support the workflow-speed promise rather than define the product. The PRD should therefore keep resume and visibility features simple and resist scope that turns the product into a broader control plane before the workflow-replacement hypothesis is proven.

## Alternatives Considered

### Alternative 1: Continuity-First

- **Description**: Make richer session memory, deeper resume behavior, and stronger continuity features the main product promise.
- **Pros**: Stronger long-term differentiation; more compelling trust story; could create a more defensible session record over time.
- **Cons**: Pulls V1 toward more product surface and more feature depth before the fast workflow loop is proven.
- **Why rejected**: The user explicitly prioritized faster project-based workflow over richer memory in the first release.

### Alternative 2: Supervision-First

- **Description**: Make session activity visibility, attention states, and monitoring cues the primary V1 story.
- **Pros**: Aligns with market demand for trust and oversight; could differentiate against simpler terminal wrappers.
- **Cons**: Risks underinvesting in the simpler daily loop of opening, switching, and resuming work; may feel heavier than the intended wedge.
- **Why rejected**: The market need is real, but the chosen wedge is workflow replacement through speed and project context, not monitoring depth.

## Consequences

### Positive

- Keeps the PRD aligned with the clearest user-stated differentiator.
- Strengthens the chance of early adoption by focusing on a high-frequency workflow.
- Preserves a tight MVP boundary and lowers the risk of V1 becoming too broad.
- Makes the product easier to explain as a calmer, more native alternative to broader ADEs.

### Negative

- V1 may feel less differentiated on trust and continuity than a deeper session-control product.
- Some users may want stronger supervision features sooner than the PRD includes.
- The product could be perceived as "mostly workflow polish" unless the core loop feels meaningfully faster in practice.

### Risks

- **Weak perceived differentiation** — Mitigate by making project switching, default context, and restore visibly better than the current workaround.
- **Trust concerns surface early** — Mitigate by preserving basic restore and enough visible session state to support confident returns.
- **Pressure to add breadth too early** — Mitigate by treating orchestration, remote flows, and broader IDE features as later-phase work unless adoption data proves otherwise.

## Implementation Notes

This ADR sets the product framing for the PRD. It intentionally defines the business priority and scope emphasis without prescribing implementation details.
