# ADR-004: Embed Full libghostty Surfaces Inside the App

## Status

Accepted

## Date

2026-05-28

## Context

V1 requires terminals that live inside the ADE window so project-scoped tabs, active context cues, and close behavior can be managed as one product surface. The repo has no existing terminal integration, and Ghostty offers multiple possible paths: full `libghostty`, a community Swift package wrapper, or driving the standalone Ghostty application externally.

Official references show that full `libghostty` is the supported embedded terminal path on macOS, but it remains public-alpha and requires AppKit `NSView` hosting with host-managed tabs, restoration, and surface inheritance. The decision is whether to prioritize direct control or rely on a third-party wrapper or external app integration.

## Decision

V1 will embed full `libghostty` surfaces inside the app and wrap them in app-owned terminal host components.

The app will:

- initialize one Ghostty app context per process
- create one terminal surface per visible tab
- host surfaces inside AppKit views/controllers owned by the app
- use host-managed tab creation, close confirmation, focus, and restore logic
- pin Ghostty to a known upstream revision and isolate integration details behind a small adapter layer

This gives V1 direct control over tab lifecycle, working-directory defaults, session shortcuts, and close semantics without splitting the user experience across multiple apps.

## Alternatives Considered

### Alternative 1: Community libghostty-spm Wrapper

- **Description**: Use a community-maintained Swift package that wraps Ghostty for faster adoption.
- **Pros**: Faster initial integration; more ready-made Swift-facing APIs.
- **Cons**: Adds third-party maintenance risk; hides important lifecycle details; may diverge from upstream behavior.
- **Why rejected**: V1 needs direct control over a core product dependency, and the wrapper would create extra uncertainty in a greenfield app.

### Alternative 2: Drive the External Ghostty App

- **Description**: Keep Ghostty as a separate application and coordinate it from the ADE.
- **Pros**: Avoids embedded terminal complexity; leverages the existing app.
- **Cons**: Breaks the integrated sidebar/session/tab experience; weakens restore and active-context guarantees; complicates native product flow.
- **Why rejected**: The PRD depends on an in-app terminal experience, not cross-app coordination.

## Consequences

### Positive

- Preserves a single native window model for project, session, and terminal context.
- Keeps tab, restore, and close behavior under app control.
- Avoids dependency on a community wrapper for the product's most important runtime surface.

### Negative

- Accepts upstream API instability and more integration responsibility.
- Requires explicit handling for AppKit events, focus, scaling, and teardown.
- Increases the burden of upgrade testing when Ghostty revisions change.

### Risks

- **Upstream libghostty API churn** — Mitigate by pinning a specific revision and hiding the C interface behind a small Swift adapter.
- **Restore limitations in upstream examples** — Mitigate by treating restoration as app-owned metadata/state reconstruction rather than relying on Ghostty to restore full live sessions.
- **Terminal host bugs affect core workflow** — Mitigate by adding focused integration tests for surface creation, tab lifecycle, and close behavior.

## Implementation Notes

The adapter should expose only the minimum V1 needs: surface creation, resize/focus hooks, close/exit state, working-directory launch config, and inherited tab creation.
