# ADR-003: In-App Modal Settings Host for Config Customization

## Status

Accepted

## Date

2026-05-29

## Context

Another ADE currently ships a single `WindowGroup` rooted in `ContentView` and exposes app-level actions through `NativeMacADEApp.commands`. The product has no `Settings` scene, no preferences state, and no reusable settings container. The approved PRD requires a dedicated config modal that feels fast, guided, and tightly connected to the existing session-first workflow.

The core question is where the new settings experience should live. The main choices are an in-app modal attached to the existing window, a native macOS Settings scene, or a hybrid approach that supports both. The user explicitly selected an in-app modal for V1.

## Decision

V1 will host configuration as an **in-app modal presented from the main window**, with a new `Settings…` app command (`⌘,`) and at least one visible in-window affordance.

The settings UI will live in the `NativeMacADE` app target alongside the existing SwiftUI shell. It will reuse the current sheet-based interaction pattern already used for session rename flow instead of introducing a parallel macOS settings scene.

## Alternatives Considered

### Alternative 1: Native Settings scene

- **Description**: Add a dedicated macOS Settings or Preferences scene and make it the primary home for configuration.
- **Pros**: Matches standard macOS conventions and cleanly separates app configuration from the workspace window.
- **Cons**: Adds a second surface to coordinate, weakens the feeling that configuration belongs to the active workflow, and introduces more UI plumbing than V1 needs.
- **Why rejected**: The approved product direction emphasizes a guided control center tied directly to daily session work, which fits an in-window modal better.

### Alternative 2: Hybrid modal plus Settings scene

- **Description**: Support both an in-app modal and a macOS Settings scene that share the same content.
- **Pros**: Offers maximum flexibility and leaves room for later platform conventions.
- **Cons**: Increases surface area, duplication risk, and state-coordination complexity for limited V1 value.
- **Why rejected**: YAGNI. One guided surface is sufficient for the first release.

## Consequences

### Positive

- Reuses an existing interaction pattern in the app shell.
- Keeps configuration close to the project → session → tab workflow.
- Avoids building and coordinating a second macOS scene for V1.

### Negative

- V1 will not behave like a standard macOS preferences window.
- The settings UI will remain coupled to the main window lifecycle.
- Future expansion to a separate settings scene may require refactoring shared content out of the modal container.

### Risks

- **Discoverability risk**: users may miss the feature if it is only in a command menu. Mitigation: provide both `⌘,` and a visible in-window entry point.
- **State-coupling risk**: modal state can sprawl inside `ContentView`. Mitigation: isolate modal presentation state and section view models instead of folding all settings logic into the main view.

## Implementation Notes

- Add the presentation trigger in `NativeMacADEApp.commands` and route visible entry from the main shell.
- Keep section-specific state and bindings outside the main session list and tab host logic.
