# ADR-004: SQLite-Centered Global Preferences Layer

## Status

Accepted

## Date

2026-05-29

## Context

Another ADE already persists workspace metadata through a single `WorkspacePersistenceStore` abstraction with SQLite and in-memory implementations. That seam currently stores projects, sessions, tabs, session shortcuts, and the restore snapshot. The approved feature introduces the first global, app-wide preferences layer for settings such as theme selection, default agent profile, and top-level keybinding overrides.

The primary design choice is whether to keep preferences inside the existing persistence boundary or split them into a separate mechanism such as `UserDefaults` or a new settings store. The user selected a SQLite-centered approach.

## Decision

V1 will use a **SQLite-centered preferences layer** by extending `WorkspacePersistenceStore` with a narrow global `AppPreferences` record and backing it with a new single-row `app_preferences` table.

The initial preferences payload will stay intentionally small and cover only the approved V1 needs: selected theme, default agent profile, and serialized top-level keybinding overrides. Agent profiles themselves will remain in `session_shortcuts`. Sensitive references will continue to stay outside SQLite.

## Alternatives Considered

### Alternative 1: Split storage with UserDefaults or AppStorage

- **Description**: Persist theme and keybinding preferences in platform defaults while keeping agent profiles in SQLite.
- **Pros**: Simple for UI wiring and familiar for app preferences.
- **Cons**: Splits the trust and testing model across two stores, complicates in-memory testing, and makes persistence behavior less uniform.
- **Why rejected**: The codebase already has one clear persistence boundary; keeping V1 inside it reduces accidental fragmentation.

### Alternative 2: New dedicated settings store

- **Description**: Introduce a second persistence abstraction just for configuration.
- **Pros**: Could isolate settings concerns and allow future expansion.
- **Cons**: Adds new abstractions, more dependency wiring, and more migration surface before the feature has proven value.
- **Why rejected**: YAGNI. The current persistence seam can support the required scope with less new infrastructure.

## Consequences

### Positive

- Keeps persistence behavior consistent across live and in-memory environments.
- Makes the new settings model testable through the same store seams already used elsewhere.
- Preserves the local-first, single-boundary design of the existing app.

### Negative

- Requires a schema migration for the first global preferences table.
- Adds more responsibility to the existing workspace metadata store.
- Makes purely UI-facing preferences part of the core persistence layer.

### Risks

- **Migration risk**: an incorrect migration could affect startup or restore. Mitigation: add focused migration tests and keep the new table independent from existing workspace tables.
- **Scope inflation risk**: the preferences record could accumulate unrelated settings. Mitigation: keep the V1 record minimal and add fields only when tied to approved product behavior.

## Implementation Notes

- Extend both `SQLiteWorkspaceMetadataStore` and `InMemoryWorkspacePersistenceStore`.
- Keep secrets or credentials out of `app_preferences`; continue using indirect references where needed.
- Favor a single-row shape over a generic key-value table for V1 clarity and type safety.
