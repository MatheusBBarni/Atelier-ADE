# ADR-001: Agent-First Scope for Config Modal Personalization

## Status

Accepted

## Date

2026-05-29

## Context

Another ADE currently has no settings or preferences surface. Personalization is limited to built-in `SessionShortcut` launch profiles for Claude and Codex, while themes, color scheme, and keyboard shortcuts are hard-coded.

The proposed feature bundles four user-visible asks into one config-modal expansion: editable keyboard shortcuts, more themes including light themes, editable default commands for Claude and Codex, and support for additional agents and commands such as OpenCode.

Market research shows that theme and shortcut customization are expected table stakes in developer tools, while agent workflow control is increasingly central to AI-native products. The product also targets solo AI-heavy power users who want predictable, supervised control. The council therefore evaluated whether V1 should be a broad personalization release, a narrow agent-management wedge, or a hybrid path.

## Decision

V1 will be scoped as an **agent-first personalization release**.

The settings surface should make agent management the primary story: users can edit built-in agent defaults, add supported agent profiles such as OpenCode, and manage command templates from one coherent configuration flow.

A **narrow shared preferences layer** should be introduced so this release does not create isolated settings paths. Theme selection and light-theme support may be included in V1 only if they cleanly reuse that shared layer and do not displace the agent-management core. Full keyboard shortcut remapping is deferred from V1 unless later evidence shows it is necessary for the release to feel complete.

## Alternatives Considered

### Alternative 1: Broad complete personalization release

- **Description**: Ship themes, light themes, keyboard shortcut editing, default-command editing, and multi-agent management together as one fully featured config modal.
- **Pros**: Stronger first impression, clearer parity with mature developer tools, more obviously complete settings story.
- **Cons**: Requires a new preferences architecture, broader UI surface, higher migration risk, and greater chance of diluting the product wedge.
- **Why rejected**: The council judged that this breadth creates too much architecture and delivery risk before the highest-value personalization behaviors are proven.

### Alternative 2: Narrow agent-management only release

- **Description**: Extend `SessionShortcut` for editable defaults and new agent profiles, but do not introduce broader settings structure yet.
- **Pros**: Fastest path, strongest focus, lowest immediate implementation risk.
- **Cons**: Leaves no coherent place for adjacent personalization, risks future fragmentation, and may feel underpowered relative to user expectations.
- **Why rejected**: The council agreed that V1 still needs a coherent settings story, even if that story stays tightly centered on agent management.

## Consequences

### Positive

- Keeps the feature aligned with Another ADE's session-first, agent-aware product identity.
- Uses the strongest existing seam in the codebase: `SessionShortcut` and default command handling.
- Creates room for theme support without committing to a broad settings platform.
- Preserves scope discipline while still improving personalization for power users.

### Negative

- Full keyboard shortcut customization is likely delayed, even though some power users will expect it.
- The release may feel intentionally incomplete if theme support is minimal or if shortcut editing is deferred.
- The team must define a narrow preferences layer carefully to avoid accidental platform creep.

### Risks

- **Scope creep**: themes and shortcuts could grow into a second product inside the feature. Mitigation: keep agent management as the headline metric and gate secondary scope tightly.
- **Trust boundary expansion**: editable commands and new agents increase the risk of unsafe or confusing launch behavior. Mitigation: require provenance, validation, and explicit user intent for custom commands.
- **Fragmented follow-on design**: if the shared preferences layer is too thin, future personalization work may still fork. Mitigation: define a single storage and UI entry point now, even if V1 only exposes a small number of settings.

## Implementation Notes

- Reuse `SessionShortcut` and `DefaultWorkspaceCommandService` as the starting point for editable built-in defaults and new built-in agent profiles.
- Add a dedicated settings/configuration surface instead of hiding personalization inside the session flow.
- Keep secrets outside SQLite and continue using indirect references where credentials are needed.
- Treat theme support as shell-level personalization; do not entangle it with command execution rules.
