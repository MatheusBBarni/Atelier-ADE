# ADR-006: Limited Keybinding Scope and Immediate Preference Application

## Status

Accepted

## Date

2026-05-29

## Context

Another ADE currently hard-codes top-level app commands directly in `NativeMacADEApp` and has no central registry for keyboard actions. The approved PRD includes keyboard customization in V1, but the earlier product ADR explicitly warns against exhaustive remapping. The updated V1 scope keeps a narrow managed command set, but it now targets the workflow controls the user asked for: tab navigation, session navigation, session search, terminal zoom, right-sidebar toggle, and settings access. Theme and keybinding changes still apply immediately, while agent default changes affect only future session creation.

## Decision

V1 will introduce a **small managed keybinding registry** with stable command IDs and default bindings for:

- `previousTab` — `⌘[`
- `nextTab` — `⌘]`
- `previousSession` — `⌘↑`
- `nextSession` — `⌘↓`
- `searchSessions` — `⌘P`
- `zoomInTerminal` — `⌘+`
- `zoomOutTerminal` — `⌘-`
- `toggleRightSidebar` — `⌘L`
- `openSettings` — `⌘,`

`toggleRightSidebar` is included now so the binding stays stable even though the right sidebar itself is still deferred.

Theme and keybinding changes will apply immediately to the running UI. Changes to the saved default agent profile will apply only to sessions and tabs created after the change; existing sessions keep their already-persisted launch profile intent.

## Alternatives Considered

### Alternative 1: App plus in-view actions

- **Description**: Centralize both app commands and selected view-local shortcuts in one registry.
- **Pros**: Broader user value and fewer hard-coded shortcuts left behind.
- **Cons**: Requires more action routing, more identifiers, and more UI coordination than V1 needs.
- **Why rejected**: The user chose a narrower V1 scope, and the codebase has no existing general action-ID layer.

### Alternative 2: Broad remapping system

- **Description**: Build a general shortcut action system for current and future commands.
- **Pros**: Strong long-term flexibility and more complete parity with mature tools.
- **Cons**: Adds broad architecture, naming, persistence, and QA scope across the app.
- **Why rejected**: Too much infrastructure for a feature whose approved V1 scope is intentionally guided and constrained.

### Alternative 3: Apply all changes only on relaunch

- **Description**: Persist settings immediately but defer visible effect until restart.
- **Pros**: Simplifies live update behavior.
- **Cons**: Feels sluggish, weakens user trust, and undercuts the product promise that configuration improves daily flow right away.
- **Why rejected**: Immediate feedback is important for themes and keyboard shortcuts, while existing session launch intent should remain stable.

## Consequences

### Positive

- Keeps V1 shortcut architecture tightly bounded.
- Aligns runtime behavior with existing restore semantics and persisted launch intent.
- Gives users immediate feedback for the most visible UI settings.

### Negative

- Many in-view shortcuts outside the managed command set will remain fixed in V1.
- The app will temporarily use two shortcut concepts: launch profiles and keyboard bindings.
- The reserved right-sidebar binding may temporarily point to a placeholder or inert action until that surface ships.
- Future expansion beyond the managed command set will require additional routing work.

### Risks

- **Expectation mismatch**: users may assume all shortcuts are editable. Mitigation: explicitly scope V1 UI and copy to the managed command set only.
- **Runtime inconsistency**: mixed immediate versus future-only behavior can confuse users. Mitigation: document the behavior in the modal and keep labels precise.
- **Deferred-surface confusion**: users may trigger the right-sidebar binding before that sidebar exists. Mitigation: document the placeholder behavior clearly and keep the command ID stable for future activation.

## Implementation Notes

- Model keyboard bindings by stable managed command identifiers rather than view-local closures.
- Keep existing session and tab records immutable with respect to previously selected launch profiles.
- Validate the live-update path with integration-heavy tests around persistence, command service behavior, and theme application.
