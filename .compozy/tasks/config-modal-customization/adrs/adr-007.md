# ADR-007: Centralize New-Session Bootstrap in WorkspaceCommandService

## Status

Accepted

## Date

2026-05-29

## Context

Another ADE currently has split semantics for session creation. If a session is created with a `SessionShortcut`, `DefaultWorkspaceCommandService.createSession` persists the session and its first tab together. If a session is created without a shortcut, the app shell creates the session first and then immediately calls `createTab` from `ContentView`. That split makes plain-session bootstrap a UI concern, while shortcut-backed bootstrap is a service concern.

The config customization feature introduces a saved default agent profile for new sessions. If the service silently resolves that default inside `createSession(projectID:shortcutID: nil)` while the UI still creates a second first tab afterward, the app can create duplicate tabs and inconsistent launch behavior. The bootstrap contract needs to become explicit and uniform.

## Decision

`WorkspaceCommandService.createSession` will become the **single owner of first-tab bootstrap for all new sessions**.

For V1, session creation will follow one path:
- if the caller passes an explicit `shortcutID`, use it;
- else if `AppPreferences` defines a saved default agent profile, use that;
- else create a plain shell-backed first tab.

The UI will stop calling `createTab` immediately after plain session creation. Existing sessions remain unchanged when preferences change; only newly created sessions use the updated default profile.

## Alternatives Considered

### Alternative 1: Keep split session bootstrap semantics

- **Description**: Leave plain sessions as a two-step UI flow and shortcut-backed sessions as a service-level flow.
- **Pros**: Avoids changing the existing session-creation contract.
- **Cons**: Makes default-profile resolution brittle, duplicates bootstrap logic, and creates double-tab risk when defaults are introduced.
- **Why rejected**: The new feature requires one authoritative place to resolve launch intent for new sessions.

### Alternative 2: Add a separate start-session API

- **Description**: Keep `createSession` as-is and introduce a second command such as `startSession` or `createSessionWithBootstrap`.
- **Pros**: Preserves the old API while making the new behavior explicit.
- **Cons**: Adds overlapping service methods and more branching in the app shell.
- **Why rejected**: YAGNI. The existing `createSession` call already owns the right concept and should be normalized instead of duplicated.

## Consequences

### Positive

- Unifies default, explicit-profile, and plain-shell session creation.
- Eliminates immediate follow-up tab creation from the UI layer.
- Makes configured-session metrics and launch-intent logging consistent.

### Negative

- Changes the behavior contract of `createSession` and requires updates in existing UI call sites and tests.
- Couples first-tab creation to session creation for all V1 flows.

### Risks

- **Regression risk**: session creation paths may change unexpectedly if any caller still assumes `createSession` returns a tabless session. Mitigation: update all call sites and add regression tests for plain, explicit-profile, and default-profile flows.
- **Preference-staleness risk**: a deleted default profile could leave a dangling preference. Mitigation: clear invalid default-profile references during delete and load paths.

## Implementation Notes

- Update app-shell helpers so session creation no longer immediately calls `createTab`.
- Preserve existing `createTab(sessionID:)` behavior for later tabs only.
- Keep restore behavior unchanged because restored tabs already persist concrete launch intent.
