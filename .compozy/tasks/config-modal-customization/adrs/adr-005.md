# ADR-005: Reuse SessionShortcut as the Curated Agent Profile Model

## Status

Accepted

## Date

2026-05-29

## Context

The codebase already has a persisted `SessionShortcut` model with label, launch command, launch arguments, secret reference, and built-in state. `DefaultWorkspaceCommandService` loads these records, seeds built-ins for Claude and Codex, and uses them to create shortcut-backed sessions and tabs. The PRD extends this product surface into curated agent expansion, editable built-in defaults, and a saved default profile for new sessions.

The main design choice is whether to keep building on `SessionShortcut` or introduce a new `AgentProfile` abstraction. The user selected reuse of `SessionShortcut`.

## Decision

V1 will **reuse and extend `SessionShortcut` as the canonical persisted agent profile model**.

Curated agents such as OpenCode will be added through the built-in catalog pattern already used for Claude and Codex. The settings UI may use clearer user-facing language such as “Agent Profiles,” but the core model and persistence path will stay anchored to `SessionShortcut`. Default profile resolution for new sessions will be centralized in `DefaultWorkspaceCommandService`.

## Alternatives Considered

### Alternative 1: New AgentProfile model

- **Description**: Create a distinct agent-profile entity and map it to session launch behavior.
- **Pros**: Cleaner naming and more explicit separation from keyboard shortcut terminology.
- **Cons**: Adds new schema, mapping logic, and duplicated concepts even though the existing model already captures the required launch contract.
- **Why rejected**: The current model is already the right technical seam; creating a new type would mostly rename the same behavior at extra cost.

### Alternative 2: Facade model over SessionShortcut

- **Description**: Keep `SessionShortcut` for persistence but introduce a higher-level wrapper type for settings.
- **Pros**: Could improve presentation semantics without changing persistence immediately.
- **Cons**: Adds translation layers and more moving parts without solving a real V1 technical problem.
- **Why rejected**: YAGNI. V1 benefits more from clear UI terminology than from a second domain abstraction.

## Consequences

### Positive

- Reuses the strongest existing domain and persistence seam in the codebase.
- Keeps session creation and restore behavior aligned with existing launch-profile logic.
- Minimizes new schema and service surface.

### Negative

- The name `SessionShortcut` now clashes conceptually with keyboard shortcut customization.
- Future expansion beyond curated agent profiles may eventually outgrow the current type name.
- Built-in customization semantics must be handled carefully to preserve provenance.

### Risks

- **Naming confusion**: engineers may confuse launch profiles with keyboard shortcuts. Mitigation: use “agent profile” consistently in UI and TechSpec narrative while keeping the code type unchanged.
- **Override ambiguity**: built-in versus customized state may become unclear. Mitigation: explicitly model or derive customized state and surface it in the settings UI.

## Implementation Notes

- Extend `SessionShortcut.builtInDefaults` with curated V1 agents.
- Add save or update pathways rather than replacing the launch-profile flow with a new subsystem.
- Resolve default-profile application in the command service rather than scattered UI entry points.
